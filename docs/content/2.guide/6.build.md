In development, the spec is generated on every request to `/openapi.json` so that route, validator and model changes appear immediately. This is convenient but slow — `ts-morph` parses the entire project on each call.

In production, the provider reads a pre-built `.adonisjs/openapi.json` instead. Run the ace command once per deploy to generate it.

## `openapi:generate`

```bash
node ace openapi:generate
```

This command:

1. Boots the application (so routes and providers are registered).
2. Instantiates the generator with the config from `config/openapi.ts`.
3. Walks every route, resolves request/response types and emits an OpenAPI 3.1 document.
4. Writes the document to `.adonisjs/openapi.json` (sorted alphabetically for stable diffs).

::tip
The file is intended to be committed only when you want to ship a static snapshot. Most setups regenerate it during the build and ignore it in git.
::

## Build Integration

Generate the spec as part of your build pipeline so the artifact is bundled with the deployment.

::warning
Currently the package relies on routes scanning (`@adonisjs/assembler`) and declaration (`@tuyau/core`), which only runs during dev server. So we must run dev server before building OpenAPI spec.
::

```bash
# Create dummy env needed for application boot
cp -n .env.example .env
# Generate random api key needed for the app
node ace generate:key
# Start dev server to generate routes declaration and kill it after 15s
timeout 15 node ace serve
# Generate openapi.json
node ace openapi:generate

# Build the project and copy generated spec to build folder
node ace build"
```


## Serving in Production

Once `.adonisjs/openapi.json` exists, the provider:

- Serves it directly at `endpoints.spec` (default `/openapi.json`)
- Renders the Scalar UI at `endpoints.ui` (default `/docs`) pointing at that spec
- Does **not** import `@outloud/adonis-openapi-generator` at runtime

This is why the generator is listed as an *optional* peer dependency — install it as a devDependency and your production image stays slim.

## Excluding from Production

If you do not want the docs available in production at all, disable the provider via config:

```ts
import env from '#start/env'

defineConfig({
  enabled: env.get('NODE_ENV') !== 'production',
  // ...
})
```

The provider then skips registering both the spec and UI routes.

## Troubleshooting

::warning
**`Failed to generated openapi for route: <pattern>`**

The generator could not resolve the type of one route. The original TypeScript error is attached as `cause`. Common causes:

- Controller method missing an explicit return type, or returning `any`
- Validator imported with a non-standard path that the configured `resolve` cannot map
- A union type that cannot be flattened (use `toOpenAPI` to override the schema)
::

For unsupported types, register a [custom handler](/guide/handlers) or add a `toOpenAPI` method to the class.
