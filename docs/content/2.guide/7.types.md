---
title: Frontend Types
description: Generate TypeScript types for the frontend from the OpenAPI spec using openapi-typescript.
---

The generated `openapi.json` is a contract. Point [`openapi-typescript`](https://openapi-ts.dev) at it and the frontend gets fully typed paths, parameters, request bodies and responses — no hand-written DTOs, no drift.

## Recommended Layout

In a monorepo, expose the types as their own package so multiple frontends can depend on one source of truth.

```
repo/
├── apps/
│   ├── api/                  # AdonisJS app
│   │   └── .adonisjs/openapi.json
│   └── web/                  # consumer (Nuxt / Next / Vite / …)
└── packages/
    └── sdk/                  # types-only package
        ├── openapi.d.ts      # generated
        └── package.json
```

## SDK Package

`packages/sdk/package.json`:

```json
{
  "name": "@acme/sdk",
  "type": "module",
  "version": "0.0.0",
  "exports": {
    ".": {
      "types": "./openapi.d.ts"
    }
  },
  "files": ["openapi.d.ts"],
  "scripts": {
    "generate": "openapi-typescript ../../apps/api/.adonisjs/openapi.json -o ./openapi.d.ts"
  },
  "devDependencies": {
    "openapi-typescript": "^7.13.0"
  }
}
```

Only `openapi.d.ts` ships. No runtime code, no bundling. Consumers `import type` from it.

## Wiring the Build

The AdonisJS provider only writes `.adonisjs/openapi.json` when you run `openapi:generate` (see [Build](/guide/build)). Chain both steps in the SDK package:

```json
{
  "scripts": {
    "generate:openapi": "cd ../../apps/api && ./scripts/openapi.sh",
    "generate": "openapi-typescript ../../apps/api/.adonisjs/openapi.json -o ./openapi.d.ts",
    "build": "pnpm generate:openapi && pnpm generate"
  }
}
```

Because `openapi:generate` requires a booted app, run it the same way you do in CI — usually a short-lived dev server, then the ace command:

```bash
#!/bin/sh
# apps/api/scripts/openapi.sh
cp -n .env.example .env
node ace generate:key
timeout 15 pnpm dev
node ace openapi:generate
```

Call this from the SDK build so a single `pnpm --filter @acme/sdk build` produces fresh types.

## Consuming the Types

Plain `fetch`:

```ts
import type { paths } from '@acme/sdk'

type ListUsers = paths['/users']['get']
type Response = ListUsers['responses'][200]['content']['application/json']

const res = await fetch('/users')
const users = (await res.json()) as Response
```

With [`openapi-fetch`](https://openapi-ts.dev/openapi-fetch) for a typed client:

```ts
import createClient from 'openapi-fetch'
import type { paths } from '@acme/sdk'

const api = createClient<paths>({ baseUrl: 'https://api.example.com' })

const { data, error } = await api.GET('/users/{id}', {
  params: { path: { id: '42' } },
})
```

`data` and `error` are narrowed to the exact response schemas declared in the spec. Path params, query params and JSON bodies are checked at compile time.

## Component Schemas

Reusable models (Lucid models, VineJS validators, classes with `toOpenAPI`) land under `components.schemas`. Reference them directly:

```ts
import type { components } from '@acme/sdk'

type User = components['schemas']['User']
```
